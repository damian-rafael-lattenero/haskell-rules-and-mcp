## F-01 🔳 — toolchain_status only checks PATH; doesn't see bundled binaries

**Step 1.** `ghci_toolchain_status` reports `hlint: unavailable` and
`2 of 8 tools available`. But `mcp-server/vendor-tools/hlint/darwin-arm64/hlint`
exists in the repo — I've been using it via `scripts/ci-local.sh`.

**Why it happens.** My `ToolchainStatus.hs` only calls
`findExecutable` (PATH lookup). It doesn't know about bundled
binaries under `vendor-tools/` like the TS MCP did.

**Why it matters.** A first-time user runs `ghci_toolchain_status`,
sees 6 of 8 tools unavailable, and thinks the install is broken.
The `summary` text "2 of 8 tools available" is scary.

**Fix proposal.** Two options:
1. Probe a list of known bundled locations after PATH fails, include
   results with `source: "bundled"` vs `"path"`.
2. Return a `remediation` hint per missing tool:
   `"Install with: cabal install hlint"`.

Option 2 is 10 LOC; I'll file it as issue on tracker.

## F-04 🐛 — Idempotent rule emits false-positive property for `[a] -> [a]`

**Step 5.** `ghci_suggest(function_name="reverse")` emits:

```
{"law": "Idempotent",
 "property": "\\x -> reverse (reverse x) == reverse x",
 "confidence": "medium"}
```

This property is **FALSE** for `reverse`:
- `reverse [1,2,3]` = `[3,2,1]`
- `reverse (reverse [1,2,3])` = `[1,2,3]`
- `[1,2,3] == [3,2,1]` → false

My rule matches on `a -> a` type shape without checking whether the
function has sort/normalize/canonicalize semantics. Result: any
involutive function gets flagged as idempotent, and the agent wastes
a `ghci_quickcheck` round-trip to find out.

**Fix proposal.** Two options:
1. Lower Idempotent confidence to `Low` when the type is `[a] -> [a]`
   (list functions are more often involutive than idempotent).
2. Only emit Idempotent when the function name has hint words:
   `sort`, `normalize`, `canon`, `dedupe`, `unique`, `clean`.

Option 2 is more precise but name-based heuristics are brittle.
Mixed: lower confidence + the heuristic bump when name matches.

## F-05 🟡 — Involutive + Self-inverse-on-lists are near-duplicates

**Same step.** Both rules fire for every `[a] -> [a]`:

```
{"law": "Involutive",
 "property": "\\x -> reverse (reverse x) == x"}
{"law": "Self-inverse on lists",
 "property": "\\(xs :: [Int]) -> reverse (reverse xs) == xs"}
```

Only difference is the type annotation on `xs`. For an agent,
"4 laws matched" overcounts the actual signal — 2 are the same
property.

**Fix proposal.** When a specialised rule (list-roundtrip) covers
the general rule (involutive) for the same type, only emit the
specialised one. The rule engine could have rule subsumption: rule
Y subsumes rule X if Y.matches => X.matches on a richer pattern.

## F-06 🐛 — Hint text assumes at least one High-confidence

**Same step.** Response:
`"hint": "Try the highest-confidence suggestion first via ghci_quickcheck. Total: 4 candidate law(s). High-confidence: 0."`

The instruction "try the highest-confidence first" is contradictory
when there are 0 high-confidence candidates. The agent has no
ordering signal.

**Fix proposal.** When High=0, change the message to
`"All suggestions are medium/low confidence — some may not hold. Run each with ghci_quickcheck to filter."`.

## F-07 🔳 — No High-confidence rule fires for typical cases

**Same step.** Only Monoid-identity in my catalog is marked High.
For common functions like reverse, negate, sort, the agent sees
all Medium/Low. Confidence scoring is too conservative.

**Fix proposal.** Involutive for `[a] -> [a]` should be High when
function name contains `reverse`, `neg`, `complement`. Associative
+ Commutative for `Ord a => a -> a -> a` with named `max`/`min`
should be High. Lift confidence based on name-shape co-signals.

## F-08 🐛 — ghci_batch schema says `{tool, args}` but code parses `{name, arguments}`

**Step 6.** Called `ghci_batch` with the schema-documented shape:

```json
{"actions": [{"tool": "ghci_type", "args": {"expression": "reverse"}}]}
```

Got: `"Invalid arguments: Error in $.actions[0]: key \"name\" not found"`.

**Root cause.** In `Tool.Batch.hs` I typed `baActions :: [ToolCall]`
and reused the existing `ToolCall` `FromJSON` instance from
`Mcp.Protocol` — which expects `{name, arguments}` because it's the
shape MCP itself uses for `tools/call`. But the descriptor's schema
documentation says `{tool, args}`. The caller follows the docs and
gets rejected.

**Severity.** High: this is the flagship innovation of Phase 9. First
external use of the tool fails because of a documentation/code
mismatch.

**Fix proposal.** Either:
1. Update the descriptor schema to say `{name, arguments}` — match
   MCP's tools/call shape exactly.
2. Add a custom `FromJSON` for batch's action type that accepts
   both `{tool, args}` (schema-documented, agent-friendly) AND
   `{name, arguments}` (MCP-spec-compatible).

Option 1 is cleaner and mirrors MCP protocol literally. Agent
doesn't learn a new key convention just for batch.

**Retry.** With `{name, arguments}` format: 3/3 actions ran
successfully. Sequencing works, ok/failed/skipped accounting
correct, per-action result shape is clean.

## F-09 🐛 — Unicode quotes mis-encoded in ghci_info output

**Step 6 via batch.** The `ghci_info(name="Maybe")` sub-result has:

```
"Defined in âghc-internal-9.1202.0:GHC.Internal.Maybeâ"
```

Those `â` are GHC's Unicode quotes (U+2018 / U+2019) being rendered
as Latin-1 bytes. My `Session.hs` uses `decodeUtf8Lenient` that does
`T.pack . BS8.unpack` — lossy above 0x7F. I literally wrote a TODO
for this in Phase 1: "Replace with decodeUtf8With lenientDecode in
Phase 2 when we add real Unicode assertions". Still there.

**Severity.** Medium. Output is readable but aesthetically broken.
Worse: any downstream text-matching on Unicode quotes (`'`/`'`) in
the output will fail.

**Fix proposal.** Swap `decodeUtf8Lenient` for `Data.Text.Encoding.decodeUtf8Lenient`:

```haskell
import qualified Data.Text.Encoding as TE

decodeUtf8Lenient :: BS.ByteString -> Text
decodeUtf8Lenient = TE.decodeUtf8Lenient
```

30-second change. Was a Phase 1 TODO, now a Phase 11 finding.

## F-10 🟡 — check_project response is ~30 KB for 42 modules

**Same step.** Every module's full `check_module` result is inlined
in `per_module`. Each entry has ~600 bytes of nested
`{gates, diagnostics, summary}` even when the module is green. For
a 300-module project that's ~180 KB in a single tool response —
pushes the client-side context window hard.

**Fix proposal.** Add an optional `compact: true` flag:
- When `compact=true` and `overall=true`: return only counts +
  summary, omit `per_module`.
- When `compact=true` and some failed: return only failures in
  `per_module`, skip the green ones.
- When `compact=false` (default, current behaviour): return the
  full per-module tree for debugging.

Another option: return `per_module` with `status: "ok"` as bare
strings instead of nested objects when everything is green.

## F-00 🔳 — Self-modifying MCP target is a footgun

**Before any call.** HASKELL_PROJECT_DIR points at the MCP's own
source tree. A future `ghci_refactor` call would rewrite files that
the server itself depends on at runtime. The agent (me) will remember
this; a less careful agent would corrupt its own compiler.

**Fix proposal.** `ghci_toolchain_status` or a startup warning should
flag when the target project is the MCP's own source. Alternatively,
introduce a read-only mode for self-dogfooding flows.
