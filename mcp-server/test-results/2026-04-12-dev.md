### Que funciono bien
- El loop edit→ghci_load→warningActions→fix es extraordinariamente fluido. Los warnings fueron 100% actionables.
- Typed holes como workflow de exploracion: escribir `_`, compilar, leer expected type y fits, implementar.
- QuickCheck inline: poder correr properties sin test suite es genial para desarrollo iterativo.
- ghci_scaffold: crear stubs automaticamente desde .cabal evita el error clasico de "can't find source".
- ghci_check_module: vista rapida de la API de un modulo con tipos.
- Project switching: cambiar entre proyectos sin reiniciar el MCP server.
- ghci_doc: documentacion de libreria accesible sin salir del flujo.
- El rename workflow (references → rename preview → Edit replace_all) fue efectivo.

### Friction points
- ghci_complete no encuentra nombres de paquetes que no estan cargados en la sesion GHCi actual. Si necesitas completions de QuickCheck antes de cargar el modulo que lo importa, no funciona.
- topFits en el JSON estructurado de holes[] vino vacio — los fits solo aparecen en el raw output. Hay que parsear el raw o usar ghci_hole_fits como tool separado.
- Despues de ghci_switch_project, los nombres no estan en scope hasta hacer ghci_load. No es un bug pero agrega un paso manual.
- ghci_rename no aplica los cambios — solo da preview. En HLS, F2 aplica directo. Aca hay que combinar con Edit, lo cual funciona pero es mas pasos.
- Sin ormolu/hlint instalados, ghci_format y ghci_lint son no-ops. Podrian sugerir instalacion automatica.
- El _notice sobre rules se repite en CADA tool call aun cuando las rules ya estan instaladas (skipped). Deberia detectar que ya estan.

### Bugs encontrados
- **topFits vacio en holes[]**: `ghci_load` con diagnostics devuelve holes con `topFits: []` pero el raw output SI tiene los refinement fits. El parser de holes no esta extrayendo los fits del output.
- **_notice persistente**: `ghci_setup()` reporta "Already up to date" pero subsequent tool calls siguen mostrando el _notice pidiendo correr ghci_setup(). La deteccion de rules instaladas no funciona correctamente entre llamadas.

### Que usaria un dev Haskell?
- **Junior**: Si, definitivamente. El loop automatico con warningActions es como tener un mentor que dice exactamente que arreglar. QuickCheck inline baja la barrera de entrada a property testing. La exploracion con typed holes y ghci_type/info es muy didactica.
- **Intermedio**: Si, como complemento. ghci_quickcheck inline es killer feature. El loop compile→fix ahorra mucho tiempo. Pero extraniaria las completions as-you-type de HLS y el go-to-definition con click.
- **Senior**: Parcial. Usaria ghci_quickcheck, ghci_eval, y ghci_batch para exploracion rapida. El loop automatico es menos valioso porque un senior ya tiene el ciclo internalizado. Pero para code review con agente, el flujo es excelente.
